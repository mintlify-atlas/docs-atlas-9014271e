This guide walks you through cloning the Scheduly API, configuring your environment, and making your first schedule generation request.

## Prerequisites

Before you begin, make sure you have the following installed:

- **Node.js 20.x** — [nodejs.org](https://nodejs.org)
- **PostgreSQL 14 or higher** — running locally or accessible via a connection string
- **npm** — included with Node.js

## Setup

<Steps>
  <Step title="Clone the repository">
    Clone the Scheduly API repository and install its dependencies.

    ```bash
    git clone https://github.com/JhojanRop/scheduly-api.git
    cd scheduly-api
    npm install
    ```
  </Step>

  <Step title="Configure environment variables">
    Create a `.env` file in the project root with the following variables:

    ```env
    DATABASE_URL="postgresql://USER:PASSWORD@localhost:5432/scheduly_db?schema=public"
    JWT_SECRET="your_jwt_secret_key"
    NODE_ENV="development"
    PORT=3000
    CORS_ORIGIN="http://localhost:3000"
    ```

    <Warning>
      Use a long, random string for `JWT_SECRET`. Never commit this value to source control.
    </Warning>

    Replace `USER` and `PASSWORD` with your PostgreSQL credentials. The database `scheduly_db` will be created by Prisma during the migration step.
  </Step>

  <Step title="Run database migrations">
    Apply the Prisma schema to your database and seed the initial data (days of the week).

    ```bash
    npx prisma migrate dev
    npx prisma db seed
    ```

    <Note>
      `prisma migrate dev` creates the database if it does not already exist, then applies all pending migrations.
    </Note>
  </Step>

  <Step title="Start the server">
    Start the API in development mode:

    ```bash
    npm run start:dev
    ```

    The API is now available at `http://localhost:3000/api`. Interactive Swagger documentation is available at `http://localhost:3000/docs`.

    <Tip>
      In development mode the server watches for file changes and restarts automatically.
    </Tip>
  </Step>
</Steps>

## Make your first requests

With the server running, follow these steps to register, authenticate, build your schedule data, and generate an optimized schedule.

<Steps>
  <Step title="Register a user">
    Create a new account. The `username` must be unique (max 50 characters), and the `password` must be at least 8 characters.

    ```bash
    curl -s -X POST http://localhost:3000/api/auth/register \
      -H "Content-Type: application/json" \
      -d '{
        "username": "jhojanRop",
        "password": "12345678",
        "fullName": "Jhojan Ropero"
      }'
    ```

    **Response:**

    ```json
    {
      "id": "<uuid>",
      "username": "jhojanRop",
      "fullName": "Jhojan Ropero",
      "createdAt": "2026-03-15T00:00:00.000Z"
    }
    ```
  </Step>

  <Step title="Login and get an access token">
    Authenticate with your credentials. The response returns an `accessToken` for API requests. A `refreshToken` is also set as an HttpOnly cookie automatically.

    ```bash
    curl -s -c cookies.txt -X POST http://localhost:3000/api/auth/login \
      -H "Content-Type: application/json" \
      -d '{
        "username": "jhojanRop",
        "password": "12345678"
      }'
    ```

    **Response:**

    ```json
    {
      "accessToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
    }
    ```

    Save the `accessToken` value — you'll use it as a Bearer token in all subsequent requests.

    <Note>
      The `-c cookies.txt` flag saves the `refreshToken` cookie to a file so you can use it later with the refresh endpoint.
    </Note>
  </Step>

  <Step title="Create a subject">
    Subjects represent the courses you want to schedule. Replace `<ACCESS_TOKEN>` with the token from the previous step.

    ```bash
    curl -s -X POST http://localhost:3000/api/subjects \
      -H "Content-Type: application/json" \
      -H "Authorization: Bearer <ACCESS_TOKEN>" \
      -d '{
        "name": "Calculus I",
        "code": "MATH101"
      }'
    ```

    **Response:**

    ```json
    {
      "id": "<subject-uuid>",
      "name": "Calculus I",
      "code": "MATH101",
      "userId": "<uuid>",
      "createdAt": "2026-03-15T00:00:00.000Z"
    }
    ```

    Note the subject `id` — you'll need it when generating the schedule.
  </Step>

  <Step title="Create a professor">
    Professors are associated with sections. You can later use them in optimization rules (e.g., `AVOID_PROFESSOR`).

    ```bash
    curl -s -X POST http://localhost:3000/api/professors \
      -H "Content-Type: application/json" \
      -H "Authorization: Bearer <ACCESS_TOKEN>" \
      -d '{
        "name": "Dr. Ada Lovelace"
      }'
    ```

    **Response:**

    ```json
    {
      "id": "<professor-uuid>",
      "name": "Dr. Ada Lovelace",
      "userId": "<uuid>",
      "createdAt": "2026-03-15T00:00:00.000Z"
    }
    ```
  </Step>

  <Step title="Create a section with a time slot">
    Sections are specific class offerings for a subject. Each section has a professor, a time slot, and one or more days.

    ```bash
    curl -s -X POST http://localhost:3000/api/sections \
      -H "Content-Type: application/json" \
      -H "Authorization: Bearer <ACCESS_TOKEN>" \
      -d '{
        "subjectId": "<subject-uuid>",
        "professorId": "<professor-uuid>",
        "startTime": "08:00",
        "endTime": "10:00",
        "days": [1, 3]
      }'
    ```

    **Response:**

    ```json
    {
      "id": "<section-uuid>",
      "subjectId": "<subject-uuid>",
      "professorId": "<professor-uuid>",
      "startTime": "08:00",
      "endTime": "10:00",
      "days": [1, 3],
      "userId": "<uuid>",
      "createdAt": "2026-03-15T00:00:00.000Z"
    }
    ```

    <Note>
      Days are represented as integers 1–7, where 1 = Monday and 7 = Sunday.
    </Note>
  </Step>

  <Step title="Create an optimization rule">
    Rules tell the scheduling engine how to score schedule combinations. Each rule has a `type`, a `priority` (lower number = higher priority), and optional `parameters`.

    ```bash
    curl -s -X POST http://localhost:3000/api/rules \
      -H "Content-Type: application/json" \
      -H "Authorization: Bearer <ACCESS_TOKEN>" \
      -d '{
        "type": "NO_EARLY_MORNINGS",
        "priority": 1
      }'
    ```

    **Response:**

    ```json
    {
      "id": "<rule-uuid>",
      "type": "NO_EARLY_MORNINGS",
      "priority": 1,
      "parameters": null,
      "userId": "<uuid>",
      "createdAt": "2026-03-15T00:00:00.000Z"
    }
    ```

    <Tip>
      You can create multiple rules with different priorities. See the [rules reference](/concepts/rules) for all available rule types and their parameters.
    </Tip>
  </Step>

  <Step title="Generate optimized schedules">
    Pass the subject IDs you want to schedule. The engine runs a backtracking algorithm with your rules and returns the **3 best schedule combinations**, ordered by score.

    ```bash
    curl -s -X POST http://localhost:3000/api/scheduling/generate \
      -H "Content-Type: application/json" \
      -H "Authorization: Bearer <ACCESS_TOKEN>" \
      -d '{
        "subjectIds": ["<subject-uuid>"]
      }'
    ```

    **Response:**

    ```json
    [
      {
        "score": 95,
        "sections": [
          {
            "id": "<section-uuid>",
            "subjectId": "<subject-uuid>",
            "startTime": "08:00",
            "endTime": "10:00",
            "days": [1, 3]
          }
        ]
      }
    ]
    ```

    The result is sorted by `score` descending — the first item is the best matching schedule.
  </Step>
</Steps>
